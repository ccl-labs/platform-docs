# ADR-006: PostgreSQL Operatorの選択

## Context

現場のオンプレKubernetesクラスタではPostgreSQLの管理に
Crunchy Postgres Operator（PGO）を使用している。
運用を続ける中で以下の2つの課題が顕在化した。

**ライセンス問題:**
PGO v5以降のライセンス方針変更により、商用環境での無償利用がグレーゾーンになった。
現場でメンバーから指摘が上がり、上位も「いつ商用NGになってもおかしくない」と認識しているが、
足元で直接の問題は発生していないため後回しになっている。

**安定性の問題:**
WAL（Write-Ahead Logging）関連の不整合が現場で発生しており、
運用上の課題になっていた。ただしWALの挙動については
当時PostgreSQL 13という古いバージョンを使用していたことも
影響している可能性があり、Operatorの問題なのかバージョンの
問題なのかは切り分けができていなかった。

これらの課題を受け、基盤移行プロジェクトを契機にPostgreSQL Operatorの再選定を検討した。
このポートフォリオではクラウド移行（Phase 13 EKS / Phase 14 GKE）を前提としており、
クラウド親和性も選定基準に加えて評価した。

---

## Options

### Option 1: Crunchy Postgres Operator（PGO）継続
- 現場での運用実績がある
- pgBackRestによるバックアップ機能に運用経験・ナレッジが蓄積されている
- **問題点:** ライセンス問題が解消されておらず、商用環境での継続利用にリスクが残る

### Option 2: EDB Postgres Operator
- EnterpriseDB社が開発する商用サポート付きOperator
- pgBackRestをバックアップエンジンとして使用しており、PGOからの移行でバックアップ運用のナレッジを継続活用できる
- 操作感がCrunchyに近く、移行コストが低い
- 商用サポートによりライセンス問題を根本的に解消できる
- **問題点:** 継続的なライセンスコストが発生する

### Option 3: CloudNativePG（CNPG）
- PostgreSQLコアコントリビュータが開発するOSS Operator（Apache 2.0）
- バックアップはbarman経由でS3互換ストレージに対応しており、MinIOやAWS S3をそのまま利用できる
- クラウド環境（EKS・GKE等）との親和性が高い
- **問題点:** バックアップエンジンがbarmanであり、pgBackRestの運用ナレッジを活かせない

---

## Decision

**選択はコンテキストに依存する。**

- 現場（オンプレ固定・クラウド移行なし）: **EDBが有力**
- このポートフォリオ（クラウド移行前提・コスト制約）: **CNPGを採用する**

---

## Reasons

### 現場文脈でEDBが有力な理由

1. **pgBackRestのナレッジを継続活用できる**
   バックアップは運用で特に重視される領域で、pgBackRestのトラブルシュート経験とナレッジが蓄積されている。
   EDBはpgBackRestを使用しており、Crunchyからの移行後もそのナレッジが活きる。
   CNPGはbarmanに切り替わるため、バックアップ運用の知識を積み直す必要がある。

2. **Crunchyからの移行コストが低い**
   操作感がCrunchyに近い。クラウド移行の予定がない状況でCNPGのクラウド親和性を採用理由にするのは弱く、
   移行コストの低さと商用サポートがあるEDBの方が合理的な選択になる。

3. **商用サポートでライセンスリスクを根本解消できる**
   現場で認識されているライセンスリスクを根本的に解消できる唯一の選択肢。

### ポートフォリオ文脈でCNPGを採用した理由

1. **Apache 2.0でライセンスコストが不要**
   EDBの商用ライセンスコストはこの文脈では合わない。CNPGはApache 2.0で制限なく使用できる。

2. **クラウド移行前提でOperatorを統一できる**
   Phase 13（EKS）・Phase 14（GKE）でも同じCNPGを使用できる。
   ローカルとクラウドで運用手順を統一でき、Operator切り替えに伴う再学習コストが発生しない。

3. **MinIOとのバックアップ/リストアを実際に検証した**
   barman経由でWSL上のMinIOに対してsample-backendのDBバックアップ・リストアが
   動作することを確認した。

---

## Consequences

**ポジティブ:**
- Apache 2.0でライセンスリスクなく長期運用できる
- Phase 13/14でも同じOperatorを使用でき、クラウド移行時の運用変更が少ない
- Crunchyからの移行で操作感に大きな差を感じなかった（移行コストが実際に低かった）

**ネガティブ・トレードオフ:**
- pgBackRestからbarmanへの切り替えにより、Crunchy時代のバックアップ運用ナレッジを直接活かせない
- 現場への導入を検討する場合は、クラウド移行の有無・バックアップ運用の現状・
  ライセンスコストの許容範囲によって結論が変わりうる
