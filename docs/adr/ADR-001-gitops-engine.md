# ADR-001: GitOpsエンジンの選択

## Context

現場のオンプレKubernetesクラスタでは、当初は運用ルールが整備されておらず、
`kubectl edit` や直接 `apply` によるクラスタ操作が混在していた。
誰が・いつ・何を変更したかの追跡が困難で、クラスタの「あるべき姿」が
どこにも定義されていないという状態だった。

第一段階の改善として、AzureDevOps のリポジトリにマニフェストを集約し、
Pipeline による push 型 CD を導入した。これによりデプロイ操作は自動化されたが、
**デプロイ後に誰かが直接クラスタを変更しても検知できない**という問題が残る。
Pipeline が保証できるのは「デプロイした瞬間の状態」のみであり、
その後のドリフトに対しては無力だった。

現場ではその後、基盤の移行プロジェクトを契機に、この課題を根本的に解消する
アーキテクチャとして GitOps の導入を検討した。

---

## Options

### Option 1: Pipeline ベースの push 型 CD を継続・改善する
- **Pros:** 既存の AzureDevOps Pipeline 資産をそのまま活用できる
- **Cons:** デプロイ後のドリフトを防ぐには別途 audit や定期的な diff チェックが必要になり、
  仕組みが複雑になる。「デプロイした瞬間の状態」しか保証できない。

### Option 2: ArgoCD（pull 型 GitOps）
- **Pros:** ドリフトの自動検出・自動修復が組み込み機能として提供される。
  Web UI でクラスタ状態を視覚的に確認できる。App of Apps パターンで宣言的管理が可能。
  CNCF Graduated プロジェクト（2022年）。
- **Cons:** ArgoCD 自体がクラスタ内に常駐するためリソース消費が発生する。
  ArgoCD 管理外で直接 apply すると元の状態に戻されるため、緊急時のオペレーションに注意が必要。

### Option 3: FluxCD（pull 型 GitOps）
- **Pros:** OCI Artifact（Helm Chart のコンテナレジストリからの直接参照）への対応が成熟している。
  CNCF Graduated プロジェクト。
- **Cons:** Web UI は標準では提供されない（別途 Weave GitOps UI が必要）。
  App of Apps 相当の機能は ApplicationSet で実現するが、設定の記述量が多くなる傾向がある。

---

## Decision

**ArgoCD を採用する。**

Option 1 はドリフト検出のために追加の仕組みが必要になり、解決したい課題の根本対処にならない。
Option 3 は pull 型 GitOps として同等の機能を持つが、Web UI が標準提供されず
App of Apps 相当の設定記述量が多い。運用経験のある ArgoCD の方が
導入・チーム展開のコストが低いと判断した。

---

## Reasons

1. **pull 型による drift 検出がコアな課題を解決する**
   Pipeline では「デプロイ後の改変」を防ぐ手段がなかった。
   ArgoCD はクラスタ状態と Git の差分を継続的に監視し、
   意図しない変更を即座に検出・通知できる。これは push 型では
   追加の仕組みなしには実現できない。

2. **実務での導入・運用経験がある**
   基盤移行プロジェクトで実際に ArgoCD を導入し、
   複数のマニフェストリポジトリを管理した経験を持つ。
   未知のツールを採用するリスクより、習熟済みのツールの
   信頼性を優先した。

3. **Web UI による可視化がチーム展開のコストを下げる**
   ArgoCD の Web UI はアプリケーションのデプロイ状態・同期状態・
   リソースツリーをブラウザ上で視覚的に確認できる。
   実務では PEチームでの導入後に開発チームへ展開した際、
   CLI に不慣れなメンバーでも UI を通じて状態を把握できたため
   受け入れがスムーズだった。k8s の習熟度に関わらず
   「今クラスタで何が動いているか」を共通認識にできる点は、
   組織としての運用コスト削減に直結する。

4. **App of Apps パターンでプラットフォーム全体を宣言的に管理できる**
   `bootstrap/root.yaml` を apply するだけで `platform/` 以下の
   全コンポーネントが ArgoCD の管理下に入る構成を実現した。
   新しいコンポーネントの追加も Application リソースを1つ追加するだけで
   GitOps の管理下に組み込めるため、プラットフォームの拡張性が高い。

---

## Consequences

**ポジティブ:**
- Git が Single Source of Truth になり、クラスタの「あるべき姿」が常に可視化される
- デプロイ後のドリフトを自動検出・通知できる
- クラスタ再作成時に `bootstrap/root.yaml` を apply するだけで
  全状態が復元される（DR 対応）

**ネガティブ・トレードオフ:**
- ArgoCD 自体がクラスタ内に常駐するため、リソース消費が発生する
- ArgoCD の管理外で直接 apply した場合、ArgoCD が元の状態に戻してしまうため、
  緊急時のオペレーションには注意が必要
