# ADR-003: Kubernetesマニフェスト抽象化方法の選択

## Context

現場のオンプレKubernetesクラスタでは、開発チームが管理するアプリケーション層のマニフェスト管理が統制されていない。
ベースとなるテンプレートは存在するものの、一部チームで独自改変をしているようで、詳細な状況は把握できていない。
結果として、リソース制限の設定漏れ・ラベルの不統一といった問題が発生する恐れがある。

この現場での経験から、ポートフォリオでは**最初から統一された
抽象化レイヤーを設け、正規の使い方から外れた実装を
構造的に防ぐ設計**にすることを方針とした。

---

## Options

### Option 1: Kustomize のみ
- **Pros:** Helm を必要とせず YAML だけで完結する。学習コストが低い。
- **Cons:** ベースマニフェストは誰かが一から書く必要があり、アプリごとに書き方がばらつきやすい。
  「正しいベース」からの逸脱を Kustomize 自体は防いでくれない。

### Option 2: Plain Helm（アプリごとに独立した Chart）
- **Pros:** 柔軟性が高く、アプリの特性に合わせた自由な構成が可能。
- **Cons:** アプリの数だけ Deployment テンプレートが存在することになり、
  共通ポリシーの変更（例: セキュリティポリシーの追加）が全 Chart への個別対応を必要とする。
  現場でのテンプレートはこのパターンに近く、やがて形骸化し統制が効かなくなる恐れがある。

### Option 3: Helm Library Chart
- **Pros:** テンプレートロジックを Library Chart に一元管理できる。
  アプリ開発者は `values.yaml` に値を渡すだけでデプロイでき、認知負荷が低い。
  `helm install` 不可という制約が「正規の使い方以外をさせない」ガードレールになる。
- **Cons:** Library Chart の設計品質がプラットフォーム全体に影響するため、
  変更時の影響範囲が大きい。アプリの特性が `values.yaml` で表現できない場合は
  拡張か例外対応が必要になる。

---

## Decision

**Helm Library Chart（`common-app` / `common-db`）を採用する。**

Option 1 はベースマニフェストの逸脱を構造的に防ぐ手段がなく、統制の維持が運用ルールに依存する。
Option 2 はアプリごとにテンプレートが分散するため、共通ポリシーの変更コストが線形に増大する。
Option 3 はテンプレートロジックの一元管理と構造的なガードレールを両立でき、
プラットフォームとして「正規の使い方以外をさせない」設計方針に合致する。

---

## Reasons

1. **テンプレートの一元管理によりプラットフォームの一貫性を保てる**
   Deploymentの正しい書き方（リソース制限・Probe・ラベル等）を
   `common-app`に一度定義することで、全アプリに均一に適用できる。
   Plain Helmではアプリの数だけテンプレートが存在するため、
   共通ポリシーの変更がアプリごとの個別対応を必要とする。
   Library Chartなら1か所の修正が全アプリに反映される。

2. **`helm install`不可という制約がガードレールになる**
   Library Chartは単体でインストールできないため、
   アプリChartというラッパー経由でしか使えない構造になっている。
   これはプラットフォームとして「正規の使い方以外をさせない」
   強制力であり、テンプレートの形骸化を構造的に防ぐ設計になっている。

3. **アプリ開発者の認知負荷を下げGolden Pathの入口になる**
   `sample-backend`の`values.yaml`はイメージ名・ポート・
   リソース制限など数十行で完結し、Deployment・Service・
   Ingress・ServiceMonitorが自動生成される。
   Kustomizeではベースマニフェストの構造知識が必要だが、
   Library Chart経由ではKubernetesリソースの詳細を
   知らなくてもデプロイ定義が書ける。

4. **Kyverno ポリシーとの整合性が保ちやすい**
   `common-app`はデフォルト値としてリソース制限・
   非latestタグ等のKyvernoポリシー通過条件を満たす値を持つ。
   Library Chart経由でデプロイすれば原則としてポリシー違反が
   発生しない設計になっており、ガードレールが多層に機能する。

---

## Consequences

**ポジティブ:**
- マニフェストのメンテナンス箇所がLibrary Chartに一元化され、
  共通ポリシーの変更コストが低い
- アプリ開発者がKubernetesの詳細を知らなくてもデプロイできる
  Golden Pathの入口として機能する
- Kyvernoポリシーとの整合性をプラットフォーム側で保証できる

**ネガティブ・トレードオフ:**
- Library Chartの設計品質がプラットフォーム全体に影響するため、
  `common-app`の変更は全アプリへの影響範囲を考慮して慎重に行う必要がある
- アプリの特性が`values.yaml`で表現できない場合（例：サイドカーが必要、
  カスタムリソースを追加したい等）は、Library Chartの拡張か例外対応が必要になる
- 既存アプリをLibrary Chart化するには現状のマニフェストの棚卸しと
  移行コストが発生する。技術的な難しさより「動いているものを
  触りたくない」という心理的抵抗の方が大きい課題になると想定される
